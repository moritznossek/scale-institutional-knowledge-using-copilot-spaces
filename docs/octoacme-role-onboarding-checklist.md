# OctoAcme — Role Onboarding Checklist

## Purpose
Standardize role setup and clarification for all team members joining a project, reducing ambiguity and enabling faster contribution.

## When to Use
Use this checklist when:
- Onboarding a new team member to a project
- Adding a cross-functional team member (e.g., Security Officer, QA Lead)
- Clarifying roles mid-project due to changes or new needs

## Pre-Kickoff Setup (PM/Project Sponsor)

### Define Roles & Responsibilities
- [ ] Identify all roles needed for this project (use [Personas](./octoacme-roles-and-personas.md) as reference)
- [ ] Assign named individuals to each role
- [ ] Document any shared or rotating responsibilities
- [ ] Share [RACI Matrix](./octoacme-raci-matrix.md) with team
- [ ] Clarify decision rights and approval authorities

### Create Role Context Documents
- [ ] Prepare project one-pager with success metrics
- [ ] Draft project timeline and key milestones
- [ ] List known dependencies and stakeholders
- [ ] Document initial constraints or requirements

### Prepare Team Communications
- [ ] Schedule project kickoff meeting
- [ ] Prepare role-specific onboarding materials
- [ ] Create project communication plan (channels, cadence, escalation paths)

---

## Role-Specific Onboarding

### For All Team Members
- [ ] Confirm understanding of project goals and success metrics
- [ ] Review project one-pager and timeline
- [ ] Understand their role and how it contributes to project success
- [ ] Know who to ask for help (role counterparts, PM escalation)
- [ ] Understand communication cadence and channels (standup, syncs, status reports)
- [ ] Access to project board, documentation, and collaboration tools
- [ ] Understand escalation paths for blockers or risks

### For Developers
- [ ] [ ] Clone repo and set up local development environment
- [ ] [ ] Review acceptance criteria template and current backlog items
- [ ] [ ] Understand Definition of Done and PR review process
- [ ] [ ] Know testing requirements and coverage expectations
- [ ] [ ] Identify Technical Lead for design questions
- [ ] [ ] Understand code review expectations and timeline
- [ ] [ ] Know security scanning and compliance requirements

### For Product Manager
- [ ] [ ] Review product roadmap and strategic context
- [ ] [ ] Understand customer needs and target users
- [ ] [ ] Clarify success metrics and how they'll be measured
- [ ] [ ] Identify stakeholders and communication plan
- [ ] [ ] Know backlog prioritization process
- [ ] [ ] Understand trade-off decisions and when to escalate
- [ ] [ ] Set up regular syncs with Project Manager and Technical Lead

### For Project Manager
- [ ] [ ] Review project charter and stakeholder list
- [ ] [ ] Understand timeline, milestones, and key dependencies
- [ ] [ ] Identify communication plan (status update frequency, escalation paths)
- [ ] [ ] Set up project board and tracking process
- [ ] [ ] Create initial risk register with known risks
- [ ] [ ] Schedule recurring ceremonies (planning, standups, reviews, retros)
- [ ] [ ] Know stakeholder preferences for updates and gate reviews
- [ ] [ ] Establish decision log and track action items

### For Technical Lead / Architect
- [ ] [ ] Review technical architecture and design decisions
- [ ] [ ] Understand system constraints and technical risks
- [ ] [ ] Know code quality standards and best practices expected
- [ ] [ ] Schedule time with developers for design reviews and mentoring
- [ ] [ ] Identify integration points and external dependencies
- [ ] [ ] Understand testing strategy and quality expectations
- [ ] [ ] Review security architecture and compliance requirements

### For QA / Testing Lead
- [ ] [ ] Review acceptance criteria and Definition of Done
- [ ] [ ] Understand success metrics related to quality
- [ ] [ ] Know testing scope (unit, integration, e2e, manual, etc.)
- [ ] [ ] Review test infrastructure and tools available
- [ ] [ ] Schedule sync with developers on testability and edge cases
- [ ] [ ] Understand pre-release verification and sign-off process
- [ ] [ ] Know security and compliance testing requirements

### For Scrum Master / Agile Coach
- [ ] [ ] Understand team composition and working agreements
- [ ] [ ] Confirm sprint length and ceremony schedule
- [ ] [ ] Know team's velocity baseline (if available)
- [ ] [ ] Understand escalation paths for blockers and impediments
- [ ] [ ] Identify key dependencies and cross-team interactions
- [ ] [ ] Set up sprint board and backlog tracking
- [ ] [ ] Schedule retrospectives and process improvement sessions

### For Stakeholder / Business Owner
- [ ] [ ] Review project charter and business case
- [ ] [ ] Understand success metrics and how they'll be measured
- [ ] [ ] Know approval authorities and decision gates
- [ ] [ ] Understand communication cadence (status updates, gate reviews)
- [ ] [ ] Know escalation paths for business-impacting issues
- [ ] [ ] Confirm budget/resource approval if needed
- [ ] [ ] Schedule stakeholder kickoff and alignment meeting

### For Security / Compliance Officer
- [ ] [ ] Review project scope and data handling requirements
- [ ] [ ] Understand compliance requirements (regulatory, policy, security)
- [ ] [ ] Know security scanning and validation tools/processes
- [ ] [ ] Schedule design review sessions with Technical Lead
- [ ] [ ] Confirm threat modeling and security planning approach
- [ ] [ ] Know pre-release security sign-off and verification process
- [ ] [ ] Understand incident escalation and response procedures

---

## Post-Onboarding (1-2 weeks after kickoff)

### Verify Readiness
- [ ] Check in with each team member individually
- [ ] Confirm they understand their role and responsibilities
- [ ] Address any open questions or concerns
- [ ] Verify access to tools, documentation, and resources
- [ ] Confirm they know who to ask for help

### Adjust as Needed
- [ ] Update RACI matrix if roles changed or clarifications needed
- [ ] Update project communication plan if preferences changed
- [ ] Document any process adjustments in project retrospectives

---

## Key Artifacts to Share During Onboarding

1. **Project Charter / One-pager**: Overview, goals, stakeholders, timeline
2. **[OctoAcme Personas](./octoacme-roles-and-personas.md)**: Role definitions and responsibilities
3. **[RACI Matrix](./octoacme-raci-matrix.md)**: Decision rights and accountability
4. **[Project Management Overview](./octoacme-project-management-overview.md)**: Lifecycle and principles
5. **Project Board Link**: Access to backlog, sprints, and tracking
6. **Communication Plan**: Meeting schedule, channels, escalation paths
7. **Risk Register**: Known risks and mitigation strategies
